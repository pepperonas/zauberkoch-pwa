# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Projekt: Zauberkoch 🧑‍🍳🍸 — KI-Rezept- & Cocktail-Generator

Mobile-first Web-App unter **https://zauberkoch.de**: Nutzer wählt Modus (Kochen/Cocktail), Länderküche, Geschmacksrichtungen und Constraints in einem 3-Schritt-Wizard — die Claude-API generiert ein vollständiges Rezept, das sich **live per SSE-Streaming aufbaut** (Titel → Zutaten → Schritte). Repo: `pepperonas/zauberkoch-pwa` (öffentlich, Open Source — auf der Landing-Page verlinkt). **Offene Selbstbedienungs-Registrierung** via Google; **zur Laufzeit im Admin-Panel umschaltbar** (Toggle „Offene Registrierung", `app_settings.open_signup`, überschreibt den `OPEN_SIGNUP`-Config-Default) — aus → Allowlist gatet. Invite-Codes wurden entfernt (2026-07-12).

## Tech-Stack

| Layer | Technologie |
|---|---|
| Backend | Python 3.12, FastAPI, SQLite (WAL), SQLAlchemy 2, Pydantic v2, Alembic |
| KI | Anthropic API, Modell via env `ANTHROPIC_MODEL` (Default `claude-sonnet-5`), Streaming → semantische SSE-Events |
| Frontend | React 19, Vite, TypeScript strict, TanStack Query |
| Styling | Material 3 Expressive — **handgebaut** (Design-Tokens als CSS Custom Properties, KEIN MUI/Ant) |
| Animation | Motion (framer-motion-Nachfolger), echte Spring-Physik |
| Auth | Google OAuth 2.0 (Auth Code + PKCE, server-seitig), Sessions als httpOnly-Cookies, CSRF-Schutz |
| Deployment | VPS 69.62.121.168: systemd `zauberkoch-api` (Port **8742** loopback) + statischer Build in `/var/www/zauberkoch.de/`, vHost im zentralen nginx, certbot. **KEIN Docker.** |

## Konventionen (verbindlich)

- **UI-Texte Deutsch**, Code/Kommentare/Commits Englisch. Strings NUR über `src/i18n/de.ts` (i18n-ready).
- **M3 Expressive handgebaut**: alle Farben/Shapes/Typo über Tokens in `tokens.css`. Moduswechsel Kochen↔Cocktail = Token-Morph (animiert), kein Neu-Rendern.
- **Motion**: echte Springs via Motion-Library — **keine linearen `ease-in-out`-CSS-Transitions als Animations-Ersatz**, keine Inline-Magic-Numbers. **MD3-Expressive-Spring-Tokens in `src/motion/tokens.ts`** (`fastSpatial`/`defaultSpatial`/`slowSpatial` mit `bounce`-Overshoot für spatial; `effectsFast`/`effectsDefault` ohne Overshoot für opacity/color; `staggerIn`, benannte Variant-Presets) — spatial darf überschwingen, Fades nie. `src/motion/springs.ts` (stiffness/damping) bleibt als älterer Satz gültig. Nur `transform`/`opacity` animieren, `will-change` gezielt. `prefers-reduced-motion` → schnelle Fades + kein Partikel-Burst, Pflicht auf jeder Animation. Aufgewertete Signatur-Animationen: **Favorit-Reward** (`FavoriteButton.tsx`), **Hero-Moment** beim Fertigwerden (`GeneratePage`: Settle-Pop + Reveal-Sweep + Haptik + Stagger), **Conjure-Stage** 320px + Atmung + rotierende Wartetexte. **Route-/View-Transitions — 2026-07-15 auf react-routers DATA-ROUTER migriert (authoritative Doku: `.claude/rules/motion.md`):** `createBrowserRouter`+`<RouterProvider>` (`App.tsx`/`main.tsx`), JEDE Navigation trägt `viewTransition` (`<Link/NavLink viewTransition>`, `navigate(to,{viewTransition:true})`), Browser-**und** In-App-„← Zurück" = `navigate(-1)`. react-router besitzt die History und rendert die neue Route SYNCHRON im VT-Callback (`startViewTransition(()=>flushSync(render))`) für PUSH **und POP** → **Browser-Back morpht endlich auf echtem Mobile-Chrome** (verifiziert fine+coarse). Shared-Naming via `useViewTransitionState('/rezept/id')` (statt `activeId`), Scroll via `<ScrollRestoration/>`. Kosten: Entry-Chunk +~15 kB gz (Data-Router-Runtime). **Der frühere manuelle `ViewTransitionProvider`/`go()`/`window.navigation`-Listener ist ENTFERNT** (er konnte den externen Browser-Back-Traverse nicht mit dem VT-Snapshot synchronisieren). ⚠️ **Die folgenden Absätze beschreiben teils das ALTE System und sind überholt, wo sie `go()`/`activeId`/`ViewTransitionProvider`/`intercept()` nennen** — gültig bleiben nur die CSS-Namen (`zk-shared-*`, `zk-d-*`, `zk-nav/header`-Hold, Root-Crossfade) und die Morph-Killer (Suspense-Prewarm, Query-Cache-vor-Navigation, framer-Entrance-Suppression bei aktivem Morph): Navigationen laufen über `useViewTx().go(to, {sharedId})`: RecipeCard-Klick = **PUSH** (`go('/rezept/id', {sharedId:id})`, startet die VT direkt), Bottom-Nav-Tabs (`go(to)` PUSH, ohne sharedId → nur Root-Crossfade). **Back/Forward:** die In-App-„← Zurück" (`go(-1, {sharedId})`) treibt ihren VT **selbst in `go()`** (robust, unabhängig von der Navigation API — v46 machte sie fälschlich Listener-abhängig → beide Back-Wege brachen). Der **Browser-/Maus-Zurück-Button** wird von einem **`window.navigation`-`navigate`-Listener** (Navigation API, `navigationType==='traverse'`) getrieben — `popstate` (auch capture-phase) feuert ERST NACHDEM react-router re-rendert hat (verlassene Seite/Detail-Hero weg → nichts zu benennen), das `navigate`-Event feuert VOR dem Commit (alte DOM noch da). **Browser-Back = observe-only, Hero IMMER benennen, NIE `intercept()` (gefixt 2026-07-15 via User-`?vtdebug=1`-Logs).** Der Bug war ein `fine &&`-Gate beim Hero-Benennen: echte Nutzer sind **coarse** (`fine:false`) → Hero unbenannt → keine Shared-Elemente → harter Cut, während der In-App-`go(-1)` (benennt IMMER) morphte. Fix: `if (sid!=null) flushSync(setActiveId(sid))` **ohne** Pointer-Gate + observe (kein `intercept()` — das unterdrückt `popstate` → old===new → Cut). Verifiziert: morpht auf fine **und coarse** (Pixel-5). **NIE wieder einbauen:** `fine`-Gate, `intercept()` auf Touch (killt Morph — Predictive-Back-Animation ist der kleinere Preis), „INTERCEPT+REDRIVE" (`navigate(replace)` im Handler wirft auf echtem Chrome + `forcedScroll`-Leak zerschoss den In-App-Back). **⚠️ „Title-Bar flackert auf Mobile":** die **App-Kopfzeile** (`.shell__header`, `zk-header`) sprang beim Zurücknavigieren, weil sie `position: sticky` war und der VT-Snapshot-Containing-Block auf Mobile den URL-Bar-Bereich mit abdeckt (Chrome toggelt die Adressleiste zu VT-Beginn → Sticky-Header re-ankert → gehaltener Snapshot springt). **Gefixt 2026-07-15: Header ist `position: fixed`** (viewport-fest, im Snapshot-Block stabil), `.shell` reserviert `padding-top: var(--header-h)`; NIE zurück auf `sticky` (Details `.claude/rules/motion.md`). Die **Browser-Adressleiste selbst** bleibt Browser-Chrome → nur via PWA-Install ganz weg. „VT auf Touch abschalten" killt den Morph → verboten. **Doppel-VT-Guard:** `go(-1)` setzt `skipTraverse`, damit der Listener den von ihm ausgelösten Traverse ignoriert. Fallback ohne Navigation API (Firefox/älteres Safari): In-App-Button morpht weiter, nur der Browser-Button ist un-animiert. **Shared-Element-Container-Transform:** Gericht-**Grafik** (`RecipeMotif style={{viewTransitionName}}`) + **Titel** morphen Position/Größe zwischen Karte und Detail-Hero. Feste Namen `zk-shared-motif`/`zk-shared-title` (`viewTransition.tsx`), gesetzt **nur** am aktiven Element (`activeId === recipe.id`) → pro Snapshot eindeutig, löst automatisch den „viele Items"-Perf-Punkt (nur die geklickte Karte wird benannt). CSS in `base.css`: Theme-Reveal-Regeln auf `.zk-theme-vt` gescopet (sonst würgen sie den Route-Root-Crossfade ab); `::view-transition-group(zk-shared-*)` morpht mit MD3-Emphasized `cubic-bezier(.2,0,0,1)` 360 ms; Root-Content crossfadet (`zk-vt-fade-in/out`, 320/180 ms). **Kritische Mechanik (mühsam erarbeitet):** Der VT-Callback gibt für **beide** Richtungen eine Promise zurück, die erst beim tatsächlichen React-Route-Commit resolved — über einen `pendingResolve`-Ref, den der Scroll-`useLayoutEffect` (auf `location.key`) auslöst (+ 500 ms Safety-Timeout). **KEIN `flushSync(navigate)`**: react-router v7 committet die Navigation nicht synchron (deferred), und rAF ist während der VT-Callback-Phase ausgehungert → beide Wege scheiterten (New-Snapshot = alte Seite → später Swap → Morph gecancelt). Drei weitere Fallen, die den Morph killten: (a) **Suspense** — die Ziel-Seite MUSS beim Commit synchron mit Hero da sein; eine lazy `RecipeDetailPage` suspendet einen Frame → Fallback-Snapshot ohne Hero → Cancel. Lösung: **`RecipeDetailPage` wird per verstecktem Off-Route-Mount vorgewärmt** (`{me && !'/rezept/' && <div hidden><Suspense><RecipeDetailPage/></Suspense></div>}` in App.tsx) → `React.lazy` ist vor der Navigation resolved, Route rendert synchron — OHNE die Seite (RecipeView/RecipeMotif, ~12 KB gz) in den Entry-Chunk zu ziehen (Lighthouse!). Schwere Dialoge (CookMode/ShareDialog) in RecipeDetailPage sind zusätzlich `lazy`. (b) **Query** — `RecipeCard` `ensureQueryData` VOR der Navigation (sonst Ladezustand ohne Hero), `onPointerDown`-Prefetch macht das Warten ~0. (c) **framer-Entrances** — der `riseIn` der `motion.section.hero` wird bei `isShared` unterdrückt (animiert den Motiv-Container und würgte die VT ab). **Ebenso `RecipeCard`**: die gestaffelte `riseIn`-Entrance wird unterdrückt, solange **irgendein** Morph läuft (`activeId != null`) — sonst re-mountet die Liste beim Zurück, die Karten stehen im New-Snapshot noch auf `opacity 0`/y-Offset → Morph-Ziel unsichtbar, Liste wirkt „hart neu gerendert" (genau das Browser-Back-Symptom). Fix 2026-07-14. **Perf:** kein `backdrop-filter`-Jank mehr (VT snapshottet den Header einmalig — der `.zk-page-vt`-Blur-Drop entfiel). **Scroll:** `history.scrollRestoration='manual'`; Position wird in `go()` synchron VOR der Navigation gemerkt (ein Scroll-Listener würde mit dem Vorwärts-`scrollTo(0,0)` racen und 0 speichern), Restore/Reset im `useLayoutEffect` auf `location.key` (POP → gemerkt, sonst 0). **Fallback:** `prefers-reduced-motion` oder fehlendes `startViewTransition` → normale Navigation ohne VT. Der frühere framer-`AnimatePresence`-Route-Wrapper + der tote `layoutId`-Container-Transform wurden entfernt.
